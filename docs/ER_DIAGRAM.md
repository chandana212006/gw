# ER Diagram Explanation

## Entities

- **USERS** (`user_id` PK) — admin accounts.
- **WELL** (`well_id` PK) — physical monitoring wells.
- **WATER_LEVEL** (`measurement_id` PK) — time-series readings for a well.
- **RAINFALL** (`rainfall_id` PK) — daily rainfall recorded per district.
- **ALERT** (`alert_id` PK) — auto-generated by trigger for low readings.

## Relationships

- WELL **1 — N** WATER_LEVEL  (one well → many readings)
- WELL **1 — N** ALERT        (one well → many alerts)
- RAINFALL links to WELL **logically** via `district` (no hard FK because
  districts are independent geo-entities, not a separate table here; in a
  larger system you would extract DISTRICT into its own table for full 3NF).

## Cardinality

| From         | To           | Cardinality | Notes                                |
|--------------|--------------|-------------|--------------------------------------|
| WELL         | WATER_LEVEL  | 1 : N       | FK `water_level.well_id` (CASCADE)   |
| WELL         | ALERT        | 1 : N       | FK `alert.well_id` (CASCADE)         |
| RAINFALL     | WELL         | N : N (soft)| Joined on `district`                 |
